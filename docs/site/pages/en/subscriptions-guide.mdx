Subscriptions deliver real-time notifications when Neotoma data changes. You can receive notifications via webhooks (HTTP POST with HMAC signature) or Server-Sent Events (SSE).

## Creating a subscription

### Via MCP

```json
{
  "entity_types": ["task", "issue"],
  "delivery_method": "webhook",
  "webhook_url": "https://example.com/neotoma-hook"
}
```

Call `subscribe` with the above arguments.

### Via CLI

Subscriptions are managed through the MCP `subscribe` action or the HTTP API. The CLI does not have a dedicated `subscribe` command; use `neotoma request subscribe` or the MCP tool directly.

### Filters

Every subscription must specify at least one filter. There is no firehose mode.

<ul className="list-none pl-0 space-y-2 mb-6">
  <li className="text-[15px] leading-7 text-muted-foreground"><code>entity_types</code> (array of strings): receive events for these entity types.</li>
  <li className="text-[15px] leading-7 text-muted-foreground"><code>entity_ids</code> (array of strings): receive events for these specific entities.</li>
  <li className="text-[15px] leading-7 text-muted-foreground"><code>event_types</code> (array of strings): receive only these event types.</li>
</ul>

## Webhook delivery

Webhook subscriptions receive an HTTP POST for each matching event:

<ul className="list-none pl-0 space-y-2 mb-6">
  <li className="text-[15px] leading-7 text-muted-foreground"><strong>Signing</strong>: every request includes <code>X-Neotoma-Signature-256: sha256=&lt;hex&gt;</code> computed over the JSON body with the subscription's <code>webhook_secret</code>. Verify this to confirm authenticity.</li>
  <li className="text-[15px] leading-7 text-muted-foreground"><strong>Retries</strong>: failed deliveries retry with exponential backoff (<code>1s, 5s, 30s, 5m</code>).</li>
  <li className="text-[15px] leading-7 text-muted-foreground"><strong>Circuit breaker</strong>: after 10 consecutive failures (configurable via <code>max_failures</code>), the subscription is deactivated. Reactivate by updating the subscription.</li>
  <li className="text-[15px] leading-7 text-muted-foreground"><strong>HTTPS required</strong>: webhook URLs must use HTTPS in production. <code>http://localhost</code> and <code>http://127.0.0.1</code> are allowed in development.</li>
</ul>

## SSE delivery

For real-time streaming without running a webhook server:

```
GET /events/stream?subscription_id=<id>
```

The stream uses standard SSE format (`id:`, `event:`, `data:` fields). Clients should send `Last-Event-ID` on reconnect to resume from where they left off. The server buffers up to 1000 events (configurable via `NEOTOMA_SSE_EVENT_BUFFER`).

## Managing subscriptions

### List active subscriptions

Via MCP: `list_subscriptions`

### Check subscription health

Via MCP: `get_subscription_status` returns delivery stats, failure count, and active state.

### Remove a subscription

Via MCP: `unsubscribe({ subscription_id: "<id>" })`

## Limits

<ul className="list-none pl-0 space-y-2 mb-6">
  <li className="text-[15px] leading-7 text-muted-foreground">Maximum 50 active subscriptions per user (configurable via <code>NEOTOMA_MAX_SUBSCRIPTIONS_PER_USER</code>).</li>
  <li className="text-[15px] leading-7 text-muted-foreground">SSE buffer holds 1000 events by default (range: 100 to 10000).</li>
  <li className="text-[15px] leading-7 text-muted-foreground">Webhook timeout is 10 seconds per delivery attempt.</li>
</ul>

## Peer sync

Subscriptions power cross-instance replication. When a subscription is configured with a `sync_peer_id`, matching events are routed through the peer sync delivery path instead of the generic webhook queue. Events originating from that peer are skipped to prevent loops.
