# OpenAPI Contract Flow

**Scope:** How HTTP endpoints, generated types, MCP tools, and CLI commands stay in sync with the OpenAPI spec. This is the canonical home for the "OpenAPI-first" workflow.

## Source of truth

`openapi.yaml` is the authoritative description of the HTTP API. All downstream artifacts are generated or validated from it:

- `src/shared/openapi_types.ts` — TypeScript types, generated by `npm run openapi:generate`.
- `src/shared/contract_mappings.ts` — per-`operationId` adapter mapping (MCP tool, CLI command, or infra-only).
- `tests/contract/` — parity + mapping enforcement (fails CI on drift).
- `docs/developer/mcp/tool_descriptions.yaml` and MCP `inputSchema` shapes in `src/tool_definitions.ts` — MCP surface parity.

Handlers in `src/actions.ts` are **downstream**. When handlers and spec disagree, the contract tests fail and the spec wins; fix the handler, not the spec expectation.

## Required flow for any endpoint change

Follow this order; do not skip steps:

1. **Edit `openapi.yaml`.** Add the path, operationId, request body, responses, and any query/path parameters. Reuse existing `components/schemas` where possible.
2. **Run `npm run openapi:generate`.** Regenerates `src/shared/openapi_types.ts`. Commit the regenerated file in the same change as the spec.
3. **Update `src/shared/contract_mappings.ts`.** Every `operationId` needs a row declaring `adapter: "mcp" | "cli" | "both" | "infra"` and, where applicable, `mcpTool` and `cliCommand` names.
4. **Implement / update the handler** in `src/actions.ts` (or wherever the route is registered). Use the generated types for the request and response shape.
5. **Wire MCP / CLI surfaces** if `adapter` includes them:
   - MCP: add / update `src/tool_definitions.ts` and dispatch in `src/server.ts`; update `docs/developer/mcp/tool_descriptions.yaml`.
   - CLI: add / update the command in `src/cli/index.ts`; register coverage in `tests/cli/cli_command_coverage_guard.test.ts`.
6. **Run contract tests.** Minimum: `npm test -- tests/contract/`. The relevant gates are:
   - `tests/contract/contract_mapping.test.ts` — every OpenAPI `operationId` has a mapping row.
   - `tests/contract/contract_mcp_cli_parity.test.ts` — MCP tools and CLI commands expose the same operations.
   - `tests/contract/openapi_schema.test.ts` — MCP `inputSchema` matches the OpenAPI request body.

## When does an `operationId` need a mapping row?

Every one. `contract_mappings.ts` is exhaustive; `contract_mapping.test.ts` enforces it. Use `adapter: "infra"` for endpoints not surfaced to MCP or CLI (e.g., health checks, internal debug).

## Response shape changes

Adding a new response field (for example, a `replayed: true` flag on `StoreStructuredResponse`) is a spec change, not an implementation change:

1. Add the field to `openapi.yaml` with a clear description and `optional: true` if back-compatible.
2. Regenerate types.
3. Populate the field in every code path that returns the schema (not just the one you were debugging). Inconsistent population across paths is a contract bug.
4. Add a contract or unit test that asserts the field appears where expected.

Do not gate client logic on the presence / absence of a rich response field as a proxy for state. If the distinction matters, declare an explicit field.

## Error responses

The same flow applies to error envelopes. When adding structured error fields (e.g., a `hint` on issue objects), declare the field in the relevant `components/schemas` (`ErrorEnvelope` or `StoreResolutionErrorEnvelope`), regenerate types, populate from the server, and test. See `docs/subsystems/errors.md` for the envelope taxonomy.

## Legacy-payload corpus

`tests/contract/legacy_payloads/` captures real request bodies that shipped in prior minor versions and asserts how the current build treats each. The corpus is the first line of defense against **silent validation tightenings** — changes that narrow an accepted request shape without declaring the narrowing in `openapi.yaml`.

Directory contract:

```
tests/contract/legacy_payloads/
  v0.4.x/
    <scenario>.payload.json       # Exact JSON body sent to /store, /ingest, …
    <scenario>.outcome.yaml       # Declared expectation; see shape below
  v0.5.x/
    …
  CHANGES.md                      # Per-version notes for outcome flips
  README.md                       # Contributor guide
  replay.test.ts                  # Runner; iterates every *.payload.json
```

Each `<scenario>.outcome.yaml` declares:

- `endpoint`: HTTP method + path (e.g., `POST /store`).
- `outcome`: one of `valid`, `deprecated`, `rejected`.
- `error_code` (when `rejected`): expected `error.code` or `issues[].code`.
- `hint_match` (optional, when `rejected`): substring / regex the issue's structured `hint` MUST contain.
- `notes` (optional): human context for why this payload is tracked.

`replay.test.ts` loads every fixture, issues the request against the test API, and asserts the outcome matches. When a PR causes a payload to flip from `valid` to `rejected`, CI fails and the author's options are: (a) restore compatibility, or (b) flip the fixture's `outcome` to `rejected`, populate `hint_match`, and add a one-line note to `CHANGES.md` under the relevant version. Option (b) is a breaking change and requires the supplement's Breaking changes section to name it (`.cursor/skills/release/SKILL.md` § Step 3).

The corpus pairs with the OpenAPI breaking-change diff gate (`scripts/openapi_bc_diff.js`): the diff script reports schema-level narrowings; the corpus reports request-shape-level narrowings. Both feed the release preflight (`.cursor/skills/release/SKILL.md` § Step 1) and are enforced as MUST / MUST NOT items in `docs/architecture/change_guardrails_rules.mdc`.

Closed request shapes (`additionalProperties: false` in `openapi.yaml`) are also enforced at runtime by `src/middleware/unknown_fields_guard.ts`, which rejects unknown top-level fields with `ERR_UNKNOWN_FIELD`. See `docs/subsystems/errors.md` for the error code contract.

## Agent attribution fields on response schemas

Agent identity is recorded on every durable write (observations, relationships,
sources, interpretations, timeline events) as additive keys inside the existing
provenance JSON — no schema migration required. The OpenAPI surface exposes
this contract through optional fields on the record components.

**Shape (optional keys inside existing `provenance` / `source_metadata` / `properties`):**

```yaml
components:
  schemas:
    AgentAttribution:
      type: object
      description: Agent identity attribution recorded on write-path records.
      properties:
        agent_public_key:
          type: string
          description: Thumbprint of the AAuth-verified signing key.
        agent_algorithm:
          type: string
          description: HTTP Message Signature algorithm (e.g. ES256, EdDSA).
        agent_sub:
          type: string
          description: sub claim from the agent token JWT.
        agent_iss:
          type: string
          description: iss claim from the agent token JWT.
        client_name:
          type: string
          description: MCP initialize clientInfo.name; self-reported.
        client_version:
          type: string
          description: MCP initialize clientInfo.version; self-reported.
        connection_id:
          type: string
          description: Existing OAuth connection id when the request is authenticated via OAuth.
        attribution_tier:
          type: string
          enum: [hardware, software, unverified_client, anonymous]
          description: Trust-tier classification derived from the evidence above.
```

**Surfaces that carry attribution:**

| Response schema / field                    | Carrier                    |
|--------------------------------------------|----------------------------|
| `Observation.metadata`                     | Inline AgentAttribution    |
| `RelationshipSnapshot.provenance`          | Inline AgentAttribution    |
| `Source.source_metadata`                   | Inline AgentAttribution    |
| `Interpretation.interpretation_config`     | Inline AgentAttribution    |
| `TimelineEvent.properties`                 | Inline AgentAttribution    |

Keys are optional; clients MUST treat missing keys as "no attribution available
at that tier" and fall back to `attribution_tier: "anonymous"`.

Derivation lives in [`src/crypto/agent_identity.ts`](../../src/crypto/agent_identity.ts);
request-scoped context is threaded via `src/middleware/aauth_verify.ts` and the
MCP `initialize` handler in `src/server.ts`. See
[`docs/subsystems/sources.md`](../subsystems/sources.md) § "Agent Attribution in
Provenance" for the full semantic contract and
[`docs/developer/cli_agent_instructions.md`](../developer/cli_agent_instructions.md)
`[ATTRIBUTION & AGENT IDENTITY]` for the client-facing contract.

## Agent instructions (MCP + CLI runtime instructions)

Agent-facing behavioral rules are not described by `openapi.yaml`. They live in `docs/developer/mcp/instructions.md` and `docs/developer/cli_agent_instructions.md`, kept in parity by `docs/developer/agent_instructions_sync_rules.mdc`. The spec describes *what* the API does; the instructions describe *how* an agent should call it. Keep them separate and synchronized.
