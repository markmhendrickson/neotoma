# Markdown Mirror — Filesystem View of Neotoma Records

## Scope

This document covers:
- Canonical markdown renderer family
- The filesystem mirror layout, write-through hooks, and rebuild semantics
- Optional git-backed history
- Bounded `MEMORY.md` export
- Determinism guarantees

This document does NOT cover:
- Schema definitions (see `docs/subsystems/schema.md`)
- Reducer / snapshot computation (see `docs/subsystems/reducer.md`)
- Relationship semantics (see `docs/subsystems/relationships.md`)
- The correction flow (see `docs/subsystems/observation_architecture.md`)

## 1. Purpose

Neotoma stores truth in SQLite as observations and reducer-produced snapshots. Many agents and humans prefer to read memory as files. The markdown mirror is a derived, read-only filesystem view of those snapshots, regenerated on write, so standard UNIX tools (`cat`, `grep`, `less`, `tree`) work against the knowledge graph without round-tripping through the API.

SQLite remains the only source of truth. Mirror files are derived; manual edits are overwritten on the next write. To modify data, use `neotoma edit <id>`, the Inspector Edit tab, or the existing correction API — all three share one batch-correction backend.

## 2. Canonical renderer family

`src/services/canonical_markdown.ts` exports a renderer per first-class data object kind:

- `renderEntityMarkdown(snapshot, schemaFieldOrder?, opts?)` — per-entity markdown with optional provenance footer.
- `renderEntityCompactText(snapshot, schemaFieldOrder?, opts?)` — dense variant wired into MCP `retrieve_entity_snapshot` (see `docs/specs/MCP_SPEC.md`).
- `renderRelationshipMarkdown(snapshot, opts?)` — relationship snapshot with linked entity names and provenance.
- `renderSourceMarkdown(source, opts?)` — source metadata with a pointer to `GET /sources/:id/content` for raw artifact retrieval.
- `renderTimelineDayMarkdown(date, events, opts?)` — consolidated per-day log.
- `renderSchemaMarkdown(schema, opts?)` — schema registry entry rendered as a data dictionary.

### Shared rules (determinism)

- Field order is schema-first, then alphabetical for unknown keys.
- Nested objects are emitted with sorted keys.
- `_deleted` fields are suppressed.
- Headers carry canonical identifiers, `last_observation_at` / `updated_at`, and the "Do not edit this file directly" notice.
- Provenance footers (when `opts.includeProvenance`) list observation_id, source_id, source_priority, and observed_at per field.
- Output is byte-for-byte reproducible: repeated renders with identical input produce identical output.

Display helpers `src/shared/entity_display_name.ts` and `src/shared/record_display_summary.ts` are reused for canonical names and one-line summaries, keeping index pages consistent with the rest of the system.

## 3. Mirror layout

Default root: `<NEOTOMA_DATA_DIR>/mirror/`. The layout is:

```
mirror/
  index.md                            # top-level TOC across all kinds
  entities/
    index.md                          # entity-type summary
    <entity_type>/
      index.md                        # list of entities in type (slug → display name → one-line summary)
      <slug-or-id>.md                 # per-entity snapshot + provenance
  relationships/
    index.md
    <relationship_type>/
      index.md
      <encoded-key>.md                # key = source_id/target_id/type, URL-safe encoded
  sources/
    index.md
    <source_id>.md                    # metadata + link to GET /sources/:id/content
  timeline/
    index.md
    <YYYY>/<YYYY-MM-DD>.md            # consolidated per-day log (bounded directory cardinality)
  schemas/
    index.md
    <entity_type>.md
```

### Slug rules

Entity slugs are derived deterministically from `canonical_name` (slug-case) with `-<short-id-hash>` disambiguation on collision. Unsafe path characters are encoded. Two entities with the same canonical_name produce stable, distinct filenames across rebuilds.

### Soft-deleted records

Soft-deleted entities and relationships remove their mirror files. Deleted sources remove their mirror file but raw content retention is governed by the existing source retention policy.

### Write-through hooks

Mirror regeneration fires at these points:

- **Entities**: `src/services/snapshot_computation.ts#recomputeSnapshot` after successful upsert. Batch-aware: `storeStructuredInternal` in `src/server.ts` accumulates affected `entity_id`s and flushes mirror writes once per request.
- **Relationships**: relationship create / delete handlers in `src/actions.ts` and the corresponding service.
- **Sources**: source storage path following uploads via `store` or `/sources`.
- **Timeline events**: existing timeline append path (already rides on `recomputeSnapshot`).
- **Schemas**: `register` / `update` / `update_schema_incremental` actions.
- **Offline / SQLite-direct writes**: same hooks on `src/repositories/sqlite/local_db_adapter.ts`.

All hooks are guarded by `mirror.enabled` and the per-kind `mirror.kinds` list. When a rendered file matches the current on-disk contents (content-hash check), the write is skipped to avoid spurious mtime churn and empty git commits.

## 4. CLI

See `docs/developer/cli_reference.md` for full syntax. Summary:

- `neotoma mirror enable` — initial enable + rebuild, optional `--git`, `--kinds <list>`, and `--gitignore` (append the mirror path to the enclosing git repo's `.gitignore`).
- `neotoma mirror disable` — stop write-through. Does not delete files.
- `neotoma mirror rebuild` — regenerate from SQLite. Scoping: `--kind`, `--entity-type`, `--entity-id`, `--clean`.
- `neotoma mirror status` — show config, per-kind file counts, git state.
- `neotoma mirror gitignore` — idempotently append the mirror path to the enclosing git repo's `.gitignore`. No-op when the mirror path is not inside a repo.

### Rebuild filters

`rebuild` always truncates and regenerates within the targeted scope. `--clean` removes stale files inside the scope that this pass did not produce. Without `--clean`, stale files are preserved.

### Activation offer

The mirror defaults to `enabled: false`. Activation (post-install onboarding) offers it as an opt-in step — see [**Activation step 6.6: Offer markdown mirror**](../../install.md#activation-step-66-offer-markdown-mirror-opt-in). The activation agent reads `doctor.mirror` (from `neotoma doctor --json`) for `enabled`, `inside_git_repo`, `git_repo_root`, `gitignored`, and `eligible_for_offer`; offers install only when `eligible_for_offer === true`; and, when the mirror path sits inside a git repo, follows up with a gitignore sub-prompt that runs `neotoma mirror gitignore --yes`.

The session/no-session intro banner also surfaces a one-line `Markdown mirror: disabled (enable: neotoma mirror enable --yes)` hint when mirror is off, so users who skipped activation still find the discovery surface.

## 5. Configuration

Extends `~/.config/neotoma/config.json`:

```json
{
  "mirror": {
    "enabled": false,
    "path": "<NEOTOMA_DATA_DIR>/mirror",
    "kinds": ["entities", "relationships", "sources", "timeline", "schemas"],
    "git_enabled": false,
    "memory_export": {
      "enabled": false,
      "path": "MEMORY.md",
      "limit_lines": 200
    }
  }
}
```

## 6. Optional git-backed history

When `mirror.git_enabled` is true, `neotoma mirror enable` runs `git init` on the mirror path and writes an initial `Neotoma mirror initialized` commit during rebuild. After each write batch, `git add` stages changed files and a single `git commit` captures the transaction:

```
update: <n> file(s) across <kinds> via <trigger>

entities:
  - <entity_type>/<slug> (<n> field(s))
relationships:
  - <relationship_type>/<key>
observations:
  - <observation_id> source=<source_id> priority=<p> observed_at=<ts>
trigger: <store|correct|interpret|relationships|sources|schemas>
author: <agent|user>
```

Semantics:

- One commit per write transaction, not per file. Multi-kind transactions (e.g. entity + relationship + timeline) collapse into a single commit.
- Commit author is derived from `agent_id` / `user_id` when available, else a stable fallback.
- Index-page changes ride in the same commit as the records that triggered them.
- Empty diffs skip the commit (content-hash no-op guard).
- `git log path/to/entity.md` returns the full observation history for that entity.

Concurrency is serialized through the mirror write queue to avoid interleaved commits.

## 7. `MEMORY.md` export

`neotoma memory-export` writes a single bounded markdown file suitable for agent harnesses that expect file-based memory.

- Ordering:
  - `--order importance` (default): type-weighted signal score `typeWeight × log2(observation_count + 2) × recency_decay` (half-life 30 days). Tier-1 types (`task`, `contact`, `event`, `transaction`, `product_feedback`, `business_opportunity`, `issue`, `user_persona_insight`, `life_tenets`, `architectural_decision`, `agent_decision`, `release`, `feature_unit`) rank above Tier-2 durable artifacts (`note`, `post`, `meeting_transcription`, `email_message`, `location`, `file_asset`, etc.). Chat bookkeeping types score 0.
  - `--order recency`: `last_observation_at desc`.
  - Both apply `entity_id` as a final tiebreaker so output is byte-for-byte reproducible.
- Bookkeeping filter: `conversation` and `agent_message` are excluded by default because they are noise in a memory manifest. Pass `--include-bookkeeping` to opt back in. Any `--exclude-types` list is applied on top of this.
- Per-entity content uses `renderEntityCompactText`, keeping output stable under KV-cache prefixes.
- Per-field cap: `--max-field-chars` (default 400) truncates long string values (post bodies, note content, transcripts) with a deterministic `… (<N> chars truncated)` suffix. Preserves byte-stability of the canonical renderer used by MCP because the cap is only applied when explicitly passed. Set to `0` to emit full values.
- Type filter: `--include-types contact,task,event`.
- Hard line cap: `--limit-lines` (default 200). When exceeded, the output is truncated with a `<!-- fold: N entities not shown -->` marker. Set to `0` to include every entity.
- Auto-regeneration: when `mirror.memory_export.enabled` is true, `MEMORY.md` is rewritten after each mirror batch.

## 8. Determinism guarantees

- Repeated renders of identical input produce byte-identical output.
- Mirror rebuilds from identical SQLite state produce byte-identical file trees.
- Memory export from identical SQLite state produces a byte-identical file.
- Git commits skip when diffs are empty, so no-op rebuilds do not produce history churn.

These properties back the public KV-cache-stability claim of the MCP `retrieve_entity_snapshot` text payload.

## 9. Editing

Mirror files are read-only. To modify data:

- **CLI one-shot**: `neotoma edit <id>` opens the entity snapshot as YAML in `$EDITOR`. On save, changed fields become a batch correction (one `correct()` observation per field, applied atomically). Optimistic concurrency uses `last_observation_at`.
- **Inspector**: the Edit tab on the entity detail page offers the same batch-correction backend via `POST /entities/:id/batch_correct`, with a live markdown preview from `GET /entities/:id/markdown`.
- **API**: `POST /entities/:id/batch_correct` with a `{ changes, last_observation_at }` body.

All three paths route through `src/services/batch_correction.ts#applyBatchCorrection`, which pre-validates changes against the entity schema and rejects the whole batch if any field fails validation.

## 10. Explicitly out of scope

- Free-form markdown diff → `correct()` parsing. The CLI `edit` command uses a YAML field view to avoid unparseable-diff, conflict-surfacing, and rejection-surfacing problems.
- Bidirectional mirror-file sync. Mirror writes flow DB → files only.
- Multi-writer concurrency beyond single-process serialization of mirror writes.
- Watch-mode mirror regeneration for external DB edits. The mirror assumes writes go through Neotoma.
- Observation files. Observations surface via the per-field provenance footer on entity/relationship mirror files and (when enabled) via git history on those same files.
