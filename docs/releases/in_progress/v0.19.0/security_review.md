2. **Proxy trust.** Does any new code trust `X-Forwarded-For`, `Forwarded`, `Host`, or `req.socket.remoteAddress` outside the canonical `forwardedForValues(req)` + `isLocalRequest(req)` helpers? Construct a tunnel-shaped request that defeats the trust check.
3. **Local-dev widening.** Does any new path reference `LOCAL_DEV_USER_ID`, the `assertExplicitlyTrusted` escape hatch, `NEOTOMA_TRUST_PROD_LOOPBACK`, or a `dev` / `sandbox` env shortcut in a way that could leak into production behavior? 4. **Unauth public route.** For every new Express route, confirm it is in `protected_routes_manifest.json` (auth-required) or in the runtime allow-list with a stated `reason`. List any new route here.
5. **Guest-access policy widening.** Does the diff change `assertGuestWriteAllowed`, `routeAcceptsGuestPrincipal`, or any guest-token issuer? Describe the policy delta in plain English and the entity types newly reachable.

## Findings

Reviewed against the six adversarial prompts above. The dominant new surface is the Google OAuth sign-in path introduced by #1924 (`src/services/google_oidc.ts`, `src/actions.ts:2698-2800`).

1. **Alternate-path auth — Google OAuth sign-in (#1924).** Two net-new unauthenticated Express routes: `GET /mcp/oauth/google/start` (`src/actions.ts:2698`) and `GET /mcp/oauth/google/callback` (`src/actions.ts:2714`). Both are correctly *intended* to be unauthenticated — they are the pre-auth sign-in entry point — and both fail closed via `isGoogleSigninEnabled()` (`src/services/google_oidc.ts:143`), returning 404 unless **both** `NEOTOMA_GOOGLE_CLIENT_ID` and `NEOTOMA_APPROVED_EMAILS` are set. On a default install the surface does not exist. Identity resolution in the callback is gated on `verifyGoogleIdToken`, which validates signature against Google's live JWKS, issuer, audience (configured client id), expiry, `email_verified === true`, and allowlist membership (`google_oidc.ts:257-311`) before any user is created. Covered by 24 unit tests in `tests/unit/google_oidc.test.ts`. **No finding on the auth logic itself.**

2. **Unauth public route — the two OAuth routes are absent from `openapi.yaml`, so the auth matrix never exercises them.** *(FINDING — advisory, carried as a residual risk.)* `scripts/security/protected_routes_manifest.json` is auto-derived from `openapi.yaml`; `grep google openapi.yaml` returns nothing, and `overrides` is empty. Consequence: `npm run security:manifest:check` reports "in sync (116 routes)" and G3's auth matrix passes **while silently not covering these two routes** — they are neither in the protected list nor in a runtime unauth allow-list with a stated reason. The gate is not wrong about what it checks; the routes are simply invisible to it. Exposure today is bounded (fail-closed by env, and the substantive checks are unit-tested), so this is not a release blocker, but it is a *gate-coverage* gap of exactly the class G3 exists to prevent. Recommend declaring both paths in `openapi.yaml` (or as explicit `overrides` entries with `requires_auth: false` + reason) in a follow-up patch.

3. **Local-dev widening — `local-login` no longer always resolves `ensureLocalDevUser()`.** The callback marks the browser session Google-verified via `setOAuthKeySessionCookie` and records `googleVerifiedUserBySessionToken` (`src/actions.ts:2782-2785`); `/mcp/oauth/local-login` then resolves the user from that session rather than unconditionally the dev user. This *narrows* rather than widens the surface (previously the local path always landed on the dev user), and `tests/unit/google_oidc_identity_resolution.test.ts` explicitly asserts "rejects a non-allowlisted verified email and never admits it as the dev user." **No finding.**

4. **Guest-access / AAuth downgrade.** No changes to `assertGuestWriteAllowed`, `routeAcceptsGuestPrincipal`, guest-token issuance, or AAuth tier requirements in this diff. The OAuth path issues an ordinary per-email local-auth user, not a privileged or tier-elevated principal. **No finding.**

5. **Proxy trust.** The one new `x-forwarded-host` read (`src/actions.ts:2859`) constructs a public-facing URL for display behind the tunnel; it joins six pre-existing identical reads (`:835`, `:860`, `:1742`, `:11490`, `root_landing/index.ts:178`). It does **not** feed an auth decision, and `isLocalRequest` / `forwardedForValues` are untouched — the auth-matrix run shows the XFF rejection path still firing on untrusted IPs. **No finding.**

6. **Shared-graph binding (`NEOTOMA_SHARED_GRAPH_USER_ID`) — highest-consequence new behavior.** *(FINDING — by-design, operator-gated; must be documented in the supplement.)* When set to a valid UUID, **every** allowlisted Google-verified email is bound to that single shared `user_id` (`src/actions.ts:2776`, `google_oidc.ts:170`) instead of an isolated per-email graph — i.e. the whole allowlisted team reads and writes one graph. This is the intended "hosted leads-graph" feature, is opt-in (unset → isolated per-email, unchanged), and fail-safes to `null` on a malformed UUID (test: "fail-safes to null on a malformed value"). The security-relevant property is that **the blast radius of the allowlist becomes the entire shared graph**: adding an email to `NEOTOMA_APPROVED_EMAILS` grants full read/write to all shared-graph data, not a scoped subset. Correctly implemented, but operators must understand it before enabling. Requires an explicit callout in the release supplement.

## Suggested negative tests

_Either point to a row in `tests/security/auth_topology_matrix.test.ts` / `tests/integration/tunnel_auth.test.ts` / `tests/security/security_hardening.test.ts`, or write the test cases needed before sign-off._

**Already covered** (`tests/unit/google_oidc.test.ts`, 24 tests; `tests/unit/google_oidc_identity_resolution.test.ts`, 2 tests):

- Wrong issuer, alternate bare-hostname issuer, wrong audience, expired token, signature not matching JWKS (tampered/wrong key).
- `email_verified: false` rejected; verified-but-not-allowlisted email rejected **without admitting the request**.
- Non-allowlisted verified email never admitted as the dev user (the privilege-escalation shape that matters most here).
- Nonce round-trips the next path and is **single-use**; unknown nonce returns null.
- `isGoogleSigninEnabled()` false unless both env vars set; `getSharedGraphUserId()` fail-safes to null on a malformed value.

**Recommended before the follow-up patch** (gaps, not blockers — see Finding 2):

- Add `/mcp/oauth/google/start` and `/mcp/oauth/google/callback` to `openapi.yaml` (or to manifest `overrides` with `requires_auth: false` and a stated reason) so `security:manifest:check` and the auth matrix cover them rather than skipping them silently.
- Add an auth-matrix row asserting both routes return **404** when `NEOTOMA_GOOGLE_CLIENT_ID` / `NEOTOMA_APPROVED_EMAILS` are unset — this is the property the whole fail-closed argument rests on, and it is currently only proven at the `isGoogleSigninEnabled()` unit level, not at the route level.
- Add an integration test asserting `normalizeOauthNextPath` rejects an absolute external URL supplied via `?next=` on the Google path (open-redirect regression guard). The helper is shared with the key-auth path and its contract is documented in the callback comment, but the Google entry point has no dedicated route-level assertion.

## Residual risks

- **Gate-coverage gap (Finding 2).** Until the two OAuth routes are declared, G3 reports "in sync" while not testing them. Anyone reading the green gate as "all routes covered" would be mistaken. Bounded by the fail-closed env gate and the unit coverage above; carried into v0.19.1.
- **Shared-graph blast radius (Finding 6).** With `NEOTOMA_SHARED_GRAPH_USER_ID` set, `NEOTOMA_APPROVED_EMAILS` becomes an all-or-nothing grant over the entire shared graph. No per-entity or per-type scoping exists for these principals. Operator-gated and opt-in, but the supplement must say so plainly.
- **Allowlist rotation is process, not code.** Removing an email from `NEOTOMA_APPROVED_EMAILS` prevents future sign-in but does not invalidate an already-issued OAuth key-session cookie. Session lifetime governs revocation lag. Not introduced by this release's design so much as inherited by it; worth an operator note.
