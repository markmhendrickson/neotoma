# v0.16.0 Release Supplement

> Range covered: **v0.15.0 → v0.16.0** (276 commits, 746 files changed).
> Tag: `v0.16.0` (annotated `e4fc80449`) on commit `2aab5c877` (`chore(release): bump version to v0.16.0`).
> Generated by RC audit pass on 2026-06-17 (see `docs/releases/in_progress/v0.16.0/rc_fixes_session_directive.md`).

## Summary

v0.16.0 is a large, broad release that opens up several first-class agent and operator surfaces:

- A first-party **agent harness SDK** (Node + Python) that enforces the Neotoma turn protocol by construction so custom loops get correct memory behavior without re-implementing MCP rules.
- A bundled **Inspector v2** with a refreshed home, full dashboard, advanced entity browsing/editing, design system pass, pinned-primitives + connect snippets, sandbox/mode-aware UI, mobile shell, and a content-negotiation merge with the REST API (Inspector and API share the same URL namespace now).
- **MCP tool & instruction surface growth**: `publish_rendered_page`, `neotoma_turn_summary`, `split_entity`, `audit_undeclared_fragments`, standing-rule injection at `initialize`, conversation-turn linking on issue filing, auto-loaded harness skills, `target_repo` for `submit_issue`, and a tightened store-first protocol.
- **Schema & reducer hardening**: `content_field` declarations + `MISSING_CONTENT_FIELD` warning, entity-type naming lint, `merge_array` correction must replace (not union), JSON-array string recovery, and graceful fallback when `update_schema_incremental` runs on an unregistered type.
- **Graph & relationships**: merge repoints relationship edges to the survivor (`relationships_repointed` in the merge response), deterministic pagination order, HTTP multi-hop traversal parity, materialized-liveness `list_relationships` pagination, and benchmark + index tuning for fan-out traversal.
- **Substrate events & SSE**: durable substrate-event log for restart-safe resume, gap-detection on resume, `retry:` field on stream open, SSE keepalive on `/mcp`, and at-rest encryption of the durable-log payload.
- **Operator/Ops**: SQLite `busy_timeout` (no more `SQLITE_BUSY`), `entities export` (round-trip of `entities import` for leave-and-rebuild), `--project-local` / `--safe` flags on `neotoma init`, observer JSONL issue import, rolling-main → RC → prod auto-deploy LaunchAgent, and an Inspector skinning surface for embedders.
- **Security**: `override_validation` per-field write policies on `agent_definition` entities, sandbox mode resolver + boot-time topology gating, `local_sandbox` mode wired into the auth middleware, `protected_routes_manifest.json` re-synced with new routes, and a weekly G5 probe over `NEOTOMA_PROBE_HOSTS`.
- **Foundation & policy docs**: `redlines.md` constitutional commitments, adopter dependency-stability commitments + codebase-status scorecard, substrate/application collaboration boundary, multi-tenant deployment topology, schema-compatibility policy, isolation/consistency/concurrency guarantees, R17 non-compete redline amendment, exit-rebuild protocol.
- **Cross-cutting fixes** for retrieval, schema, correction, relationship list/delete discovery, mcp/cli mechanical parity gaps, issues-sync provenance, and submit_issue body normalization (`\n` decoding).

Plus the post-RC fixes for the Windows `spawn EINVAL` regression on `.cmd` shims (#1677) and the Inspector pre-merge UI audit (#1674).

## Highlights for npm package users

### `@neotoma/agent` protocol-enforcing harness SDK (#318)

Provider-agnostic agent harness that wraps `@neotoma/client` with the canonical Neotoma turn protocol so custom agent loops get correct memory behavior without hand-implementing MCP instruction rules.

- **New package `packages/agent/`** (`@neotoma/agent`): `withMemory()` wrapper, explicit `NeotomaMemory` (`open_turn` / `close_turn`), `turn_helpers`, `turn_report`, and `diagnose` utilities.
- **Turn lifecycle by construction** — bounded retrieval from the user message, user-phase `conversation` + `conversation_message` store with `PART_OF` and `REFERS_TO`, agent invocation with `ctx.retrieved`, assistant-phase store with separate `turn_key` suffix and idempotency key.
- **Contract tests** lock store payload shapes against the live client (`tests/contract/sdk_client_store_shape.test.ts`, `tests/unit/agent_memory.test.ts`, `tests/integration/agent_memory_turn_lifecycle.test.ts`).
- **Docs** — `docs/developer/sdk_agent.md` and site page `docs/site/pages/en/sdk-agent.mdx`.

```bash
npm install @neotoma/agent @neotoma/client
```

```ts
import { HttpTransport } from "@neotoma/client";
import { withMemory } from "@neotoma/agent";

const wrapped = withMemory(yourAgentFn, {
  transport: new HttpTransport({ baseUrl, token }),
  conversationId: "conv-2026-05-20",
  platform: "my-agent",
});
```

### Python `NeotomaMemory` protocol layer (#322)

Python parity for the same store-first turn protocol — HTTP-only client reaching the Node Neotoma engine over REST.

- **New modules** in `packages/client-python/`: `memory.py`, `with_memory.py`, `helpers.py` with sync and async `with_memory` / `NeotomaMemory.open_turn` / `close_turn`.
- **Claude Code hooks** — `packages/claude-code-plugin/hooks/neotoma_client/` gains the same memory layer for Python hook plugins.
- **Tests** — `packages/client-python/tests/test_memory.py`.
- **Docs** — `docs/developer/sdk_python.md` and site page `docs/site/pages/en/sdk-python.mdx`.

```bash
pip install neotoma-client
```

```python
from neotoma_client import NeotomaClient, with_memory

client = NeotomaClient(base_url="http://127.0.0.1:3080", token="dev-local")
wrapped = with_memory(my_agent, transport=client, conversation_id="conv-1")
result = wrapped("Tell me about Acme Corp")
```

### `@neotoma/client` shape fix (#316)

`@neotoma/client` now reads entities from the top-level store response shape (`response.entities`) instead of a nested envelope. Affects any consumer that was destructuring `response.structured.entities` to work around the bug; that path still resolves but the canonical surface is now the top level.

### Bundled Inspector by default

`npm run build` now chains `build:inspector` so an installed package boots a working bundled Inspector without an extra step. Inspector and the REST API share the same URL namespace dispatched on `Accept`: browsers get the SPA shell; agents/curl/`fetch` with `Accept: application/json` get the existing API unchanged.

- **Service:** `src/services/inspector_mount.ts` + `src/services/sandbox_mode.ts`.
- **Tests:** `tests/security/inspector_content_negotiation.test.ts` (20 cases), `tests/security/sandbox_mode_resolver.test.ts` (29 cases).
- **Operator-facing routing change**: `/inspector/*` is retired; the SPA serves at `/` (with `/openapi*.yaml` and other generated routes carved out so they keep returning their canonical content type).

### MCP protocol & instruction additions

- **Standing-rule injection at `initialize`** (#194 / b24f7fb2a) — `MCP initialize` response includes `serverInfo._neotoma.standing_rules`, an array of persistent agent instructions stored as `standing_rule` entities with `title`, `rule_text`, optional `scope`, `priority`. Agents apply these from the first turn without re-querying. See `docs/developer/mcp/instructions.md` [STANDING RULES] and `docs/developer/rule_neotoma_sync.md`.
- **Auto-load skills at `initialize`** (#205) — `serverInfo._neotoma.available_skills` lists the skills installed by `neotoma setup`; harnesses can invoke them by name (`/store-data`, `/remember-codebase`, etc.) without re-discovery.
- **Conversation-turn linking on `submit_issue`** (#192 / 4eab5e8dc) — Filed issues get a REFERS_TO edge from the originating conversation_message turn so the exact turn of creation is traceable. Supported via new `conversation_turn_id` parameter on `submit_issue`.
- **`target_repo` parameter on `submit_issue`** (#946 / 11747dfcb / 04971512a) — Override the GitHub mirror destination (`owner/repo`) per-call without changing the Neotoma authoring home. Required when filing issues about a non-default repo.
- **Auto-file issues when pre-consented** (#194 / 8979becfc / d105b3c91) — Proactive issue-filing mode honors an `auto_file_issues` standing rule / preference; consent prompt is shown only on first encounter when no preference exists.
- **Store-first protocol tightening** (48a0c9b78) — Every external-tool write action (email, GitHub comment, calendar event, webhook) MUST store intent in Neotoma BEFORE executing the action. Adds entity-type mapping for common external actions and an explicit retry-safe idempotency guard. See `docs/developer/mcp/instructions.md` [STORE-FIRST PROTOCOL].
- **Per-record extraction checklist** (75f9d2425 et al.) — Mandatory entity extraction (people/orgs, temporal commitments, transactional facts, locations, GH artifacts, outreach implications) on every external-tool record, with depth-of-capture rules (hydrate via detail endpoint; persist body via unstructured path > 100 KB; idempotent upgrade via `correct`).
- **GitHub entity extraction guidance** (75d9725e4) — Canonical `[number, repo]` identity for `issue` / `pull_request`, `[external_id]` for `organization`, `[name]` for `project` extracted from email/calendar/web records. See `docs/subsystems/github_entities.md`.
- **Awaiting-reply task derivation** (5d54575d5 / #303) — Outbound `email` entities with pending-reply signals auto-create a linked `task` (`task_type: "awaiting_reply"`, `status: "pending"`) via a schema-driven derived-entity rule.
- **`conversation_message_count` in `StoreStructuredResponse`** (FU-2026-05-001 / #272) — Store response surfaces the current per-conversation message count so widgets and tools can show turn counts without a follow-up query.
- **`MISSING_CONTENT_FIELD` store warning + `content_field` schema declaration** (#949 / 16a59944f) — Schemas can declare a long-form `content_field` (e.g. `body` on `note`, `content` on `gist`). When a write omits or empties it, the server emits a non-blocking `MISSING_CONTENT_FIELD` warning so agents do not silently lose long-form bodies.

### MCP / CLI parity & retrieval fixes

- **Resolve literal `ent_…` ids in `retrieve_entity_by_identifier`** (#1561/#1550/#1597 / c23a46183) — Bypasses name/text matching for primary-key lookups; returns `match_mode: "direct"` or an explicit not-found with `hint`.
- **Honor `entity_types` (plural) filter in `retrieve_entities`** (0b9f91e8c).
- **Lightweight `entities list` projection includes `status` + `--status` filter** (#1618 / f6d408adf, 126b669c1).
- **Retrieval false-negatives swept** (#1495/#1496/#1551 / 14c3f266e), schema/unknown_fields/correct gaps (#1559/#1552/#1540/#1549/#247 / 898e7c4b4), `list_relationships`/`delete` discovery (#277 / 3af823fbd), and a batch of mechanical wins (#401/#1504/#1505/#1550/#1506/#367/#369/#370 / c0c987027).
- **Resolver duplicate surface** (#1457 / 807bbe957) — Surfaces a duplicate when a single-token name prefixes an existing multi-token canonical name; surfaced through `/audit` and Inspector entity list (#1459).

## Breaking changes

- **`/inspector/*` mount point retired** (31ddbf009) — Inspector now serves at `/` via content negotiation. Clients hardcoding `/inspector/*` should switch to the root. The early `/openapi*.yaml` SPA-shell exclusion preserves OpenAPI yaml serving (0c5a582ed).
- **No breaking changes to OpenAPI/MCP write contracts in this release.** The merge response gains an additive field; store response gains additive fields. Confirm `additionalProperties` posture remains permissive on consumer side.

## RC cycle fixes (post-rc.1)

Two additional merges landed on top of the scope above and are worth calling out in release notes.

### Inspector pre-merge UI audit (#1674)

Operator-focused Inspector polish + design-system consistency pass — covered under "Inspector v2 → Pre-merge UI audit".

### Windows `spawn EINVAL` regression fix (#1676 / #1677)

Field-reported regression: on Windows with modern Node.js (post CVE-2024-27980 patch — Node ≥ 18.20.2 / 20.12.2 / 21.7.2, all v22+), `child_process.spawn('npm.cmd', ...)` and other `.cmd` shims (`npx.cmd`, `claude.cmd`, `tsx.cmd`, `neotoma.cmd`) threw `EINVAL` unless `shell: true` was set. Most user-visible consequence: `neotoma api start` failed on Windows, so the local data API never came up and CLI writes were effectively blocked.

- **New helper** `src/shared/spawn_platform.ts` (`IS_WINDOWS`, `WIN_SHELL`, `shellOnWin()`) centralizes the `shell: true`-on-win32 decision with explicit security reasoning (args stay an argv array — no command-string concatenation, no shell-injection vector).
- **Updated spawn sites** — `src/cli/index.ts`, `src/cli/hooks.ts`, `src/cli/doctor.ts`, `src/services/schema_registry.ts`, `src/mcp_dev_shim.ts` now use the helper. `doctor.ts` also maps `which` → `where` on Windows.
- **Tests** — `src/shared/spawn_platform.test.ts` covers both platform branches and the spread-into-options shape.

## Known issues (won't ship a fix this release)

Surfaced during the RC audit; tracked for follow-up.

- **#1668 — `store`: two `agent_message` rows in one request can silently merge when `message_id` is omitted.** A single `/store` request with both a user and assistant `agent_message` (or `conversation_message`) sharing one `turn_key` and no `message_id` collapses to one entity (`matched_existing` action) instead of two distinct rows. Server-side protection (`ERR_CONVERSATION_MESSAGE_ROLE_CONFLICT`) has been in place since v0.12.0 for the standard ordering and now has additional single-batch regression coverage; if the guard does not fire in your environment, ensure you are on the latest server. **Workaround:** use the `:assistant` suffix on the closing message (`turn_key: "{conversation_id}:{turn_id}:assistant"`) — the canonical pattern the MCP turn lifecycle already documents.
- **#1598 / #1587 — auto-enhance queue / schema drift errors in server logs** (`AUTO_ENHANCE_QUEUE`). Visible in production logs as `APIError:store` events on certain entity types (e.g. `checkpoint_brief`); does not corrupt stored data — observations are preserved, but the snapshot reducer may exclude undeclared fields. Use `audit_undeclared_fragments` (MCP) or `neotoma schemas audit-fragments` (CLI) to triage; promote high-occurrence fields via `update_schema_incremental` / `register_schema` per the per-store `unknown_fields` hint.
- **#1667 — public endpoints returning 401 / tight CORS posture.** RC audit identified some endpoints that surface 401 where 200/anonymous would be more appropriate, and a tighter-than-needed CORS configuration for public read paths. Read paths intended for public consumption should continue to be inspected against the auth posture documented in `docs/subsystems/auth.md` before relying on them anonymously.
- **`agent_message` role-conflict guard is direction-asymmetric** — An unusual `[assistant, user]` batch order would slip past the in-batch state map check (the guard fires on the *second* row's role conflicting with a prior `user` row in the same batch, not the inverse). Not a v0.16.0 blocker; tracked as a small follow-up.
