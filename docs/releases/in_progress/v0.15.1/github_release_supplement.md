# v0.15.1 Release Supplement

## Summary

v0.15.1 is a patch release bundling transport, contract, and reducer fixes:

1. **MCP SSE keepalive (#1483)** — the MCP StreamableHTTP `/mcp` GET SSE stream now emits an application-level heartbeat so it survives reverse-proxy and client idle timeouts instead of silently dropping under load.
2. **`retrieve_graph_neighborhood` `entity_id` (#276)** — `related_entities` items now expose the canonical `entity_id`, with the legacy `id` field restored as a deprecated alias.
3. **`merge_array` correction replace (#1541)** — a higher-priority `correct()` on an array-typed `merge_array` field now replaces lower-priority array contributions instead of unioning with them.
4. **Schema / unknown-fields / correct cluster (#1559, #1552, #1540, #1549, #247)** — a new `describe_entity_type` MCP tool; new non-fatal store-response signals (`store_warnings[]`, `required_fields_missing[]`, `unknown_fields`) covering undeclared and missing-required fields; and a behavior flip so `correct` now accepts undeclared fields (append path) instead of rejecting them. The MCP and HTTP `correct` responses are unified to an identical shape.
5. **`delete_relationship` discovery (#277)** — `delete_relationship` now verifies the supplied `(relationship_type, source_entity_id, target_entity_id)` triple matches a live edge, returning `404` with a discovery hint instead of a silent no-op deletion; `list_relationships` excludes soft-deleted edges by default and gains an `include_deleted` opt-in.
6. **`submit_issue` `target_repo` cross-repo mirror (#946, #942)** — `submit_issue` (and `POST /issues/submit`) gains an optional `target_repo` (`owner/repo`) parameter that redirects the GitHub mirror destination for that one call, with the stored `issue` / `conversation` identity namespaced to the target repo and `owner/repo` shape validation. The Neotoma authoring home (operator instance) is unchanged.
7. **Cold-start onboarding: simpler evaluate prompt + `neotoma status` (#361, #362)** — the evaluation prompt users paste into an LLM is reduced to one sentence + the `/evaluate` link, the `neotoma doctor` CLI command is renamed to `neotoma status` (with `doctor` kept as a deprecated alias), and the `/evaluate` page is restructured to lead with declarative facts instead of a "read → run → consider" funnel, all to stop cautious LLMs from refusing before they read any docs.

## What changed for npm package users

### MCP `/mcp` SSE keepalive (#1483)

The MCP server holds a long-lived GET SSE stream open for server-initiated messages, but the SDK emits no application-level heartbeat on that stream, and Node's `keepAliveTimeout` only governs socket reuse. A reverse proxy (Cloudflare Tunnel, nginx, ngrok) or the client's own idle timeout could then silently close the idle stream mid-session — the process stayed healthy (`/health` green) but the client lost the Neotoma tool registry ("tools not registered") with no recovery short of a manual disconnect/reconnect.

- The long-lived GET SSE stream now emits a periodic SSE comment heartbeat frame (`: hb\n\n`, default every 25 s), sets `X-Accel-Buffering: no` on the `text/event-stream` response so buffering proxies flush each frame immediately, and enables a bounded TCP keepalive on the socket. Heartbeat frames are SSE comment lines, ignored by clients per the spec, and interleave safely with the SDK's own events. This keeps proxies and clients from treating the stream as idle and silently closing it.
- On by default; no configuration required.

#### New runtime configuration (server process)

Read by the Neotoma HTTP server when it serves `/mcp`. Documented in `docs/developer/cli_reference.md` § MCP / SSE transport tuning.

| Environment variable | Default | Purpose |
|----------------------|---------|---------|
| `NEOTOMA_MCP_SSE_KEEPALIVE_MS` | `25000` | Interval (ms) between SSE comment heartbeat frames on the MCP GET SSE stream. Set to `0` (or any value `<= 0`) to disable the heartbeat; an unset, empty, or non-numeric value uses the default. TCP keepalive on the socket stays enabled regardless (probe delay clamped to `[15s, 60s]`). |

(The pre-existing `NEOTOMA_KEEPALIVE_TIMEOUT_MS` / `NEOTOMA_HEADERS_TIMEOUT_MS` socket-level knobs from v0.13.0 are now also documented in the same table.)

The SDK's `retryInterval` transport option was evaluated and deliberately not used: in SDK 1.29 it only emits an SSE `retry:` field inside the priming event, which is written solely on the POST→SSE path and solely when an `eventStore` is configured. Neotoma configures no event store, and the stream that drops is the standalone GET SSE stream, so setting `retryInterval` would be an inert knob with no runtime effect.

### `retrieve_graph_neighborhood` related_entities now carry `entity_id` (#276)

- Each item in the `related_entities` array exposes the canonical `entity_id`, matching `relationships[].source_entity_id` / `target_entity_id` and every other Neotoma response surface (resolves #276). The raw database `id` column is still present as a deprecated alias with the same value for one minor release; prefer `entity_id` and stop reading `id`.
- This patch also restores the `id` field that v0.15.0 silently dropped from `related_entities` when it renamed `id` → `entity_id` (bundled, undocumented, in the #262 mirror change). Re-adding `id` un-breaks any v0.15.0 caller that depended on the legacy field.

### Schema introspection, store data-quality signals, and undeclared-field corrections (#1559, #1552, #1540, #1549, #247)

This cluster makes a known-but-unfamiliar entity type's schema discoverable before storing, surfaces (without rejecting) data-quality gaps on a store, and lets `correct` append values into fields a schema has not yet declared.

- **New MCP tool `describe_entity_type` (#247).** Returns the full active schema for one `entity_type`: `field_names`, per-field `type` / `required` / `description` (`field_summary`), `required_fields`, `schema_version`, and the raw `schema_definition`. Read-only, no persistence side effects. Call it before the first store of a known-but-unfamiliar type so the write lands with zero `unknown_fields` and zero `required_fields_missing`. Errors with `InvalidParams` when the type has no active schema. Reachable from `src/shared/contract_mappings.ts`; CLI parity via `entity-types describe`.

- **New non-fatal store-response signals.** A successful `/store` (and the MCP `store` tool) may now return:
  - `store_warnings[]` — array of `{ code, message, observation_index, entity_type, entity_id }` data-quality warnings. New codes: `UNKNOWN_FIELD` (#1552) for a field not declared on the schema, and `MISSING_REQUIRED_FIELD` (#1559) for a schema field declared `required: true` that the observation omitted or left empty.
  - `required_fields_missing[]` — array of `{ entity_type, field, observation_index }` for required fields the write left unset (#1559).
  - `unknown_fields` / `unknown_fields_count` and a conditional `hint` (#1552, #1549) for undeclared fields. None of these reject the write — the value is preserved on the observation and routed to `raw_fragments`, but the reducer projects only declared fields, so it does not surface in the entity snapshot until the field is declared. The `hint` is computed against the schema's identity configuration (#1549): when the schema declares `canonical_name_fields` it prescribes `update_schema_incremental`; otherwise it points at `correct`-ing into a declared field or `register_schema`, because `update_schema_incremental` would dead-end on `ERR_SCHEMA_MISSING_IDENTITY_CONFIG`.

- **`correct` now accepts undeclared fields (#1540) — behavior flip.** Previously `correct` rejected a `field` not declared on the schema (it threw "Unknown field for entity type"). It now mirrors `store`'s append path: the correction observation is written, the value is mirrored to `raw_fragments`, and the response returns `unknown_field: true` plus a `hint` and structured `details: { entity_type, field }`. The value is excluded from the snapshot until the field is added to the schema. Use this to attach new structured data to an existing entity without an up-front schema migration.

- **MCP ↔ HTTP `correct` response unified.** Both transports return the same body (`observation_id`, `entity_id`, `field`, `value`, `message`, and on the undeclared path `unknown_field`, `hint`, `details`) via a shared `buildCorrectionResponse`. The HTTP transport additionally merges `success` and `snapshot`. The unknown-field determination is computed by a single shared `resolveCorrectionSchema` helper, so a schema-registry IO failure now surfaces as an error on both transports instead of being silently coerced into a "declared field" outcome on HTTP. A request to `correct` whose `entity_type` has no active schema now returns `ERR_NO_SCHEMA_FOR_ENTITY_TYPE` on HTTP (parity with the MCP `InvalidParams` throw), where the HTTP path previously accepted it as an undeclared-field correction.

### `submit_issue` `target_repo` cross-repo mirror (#946, #942)

Previously `submit_issue` always mirrored to the operator's globally-configured GitHub repo (`NEOTOMA_ISSUES_REPO`). An operator running multiple repos from one Neotoma instance had no way to file an issue about a sibling or downstream repo through the canonical path — they had to bypass it (raw `gh issue create`), losing the conversation thread and `entity_id` linkage.

- **New optional `target_repo` (`owner/repo`) parameter** on `submit_issue` (MCP tool) and `POST /issues/submit` (HTTP). When set, the GitHub mirror is created in that repo for the single call; when absent, behavior is unchanged (mirror goes to `NEOTOMA_ISSUES_REPO`).
- **Identity is namespaced to the effective repo.** The stored `issue` entity's `repo` / `github_url`, the external-actor repository, the `conversation` thread id, and the body `turn_key` are all derived from the override (or the configured repo when absent), so the `github_number + repo` canonical identity points at the repo the issue actually lives in. A later `sync_issues` against the target repo reuses the same entity/conversation instead of forking a duplicate.
- **The Neotoma authoring home is unaffected.** Only the GitHub mirror destination is redirected; the canonical record still lands in the operator instance (`issues.target_url`).
- **`owner/repo` shape validation.** A malformed value (bare repo, full URL, whitespace, extra path segments) is rejected at the HTTP and MCP boundary with a structured `owner/repo` validation message, instead of silently degrading to "stored locally, no mirror".

### Cold-start onboarding: evaluate prompt, `neotoma status`, evaluate page (#361, #362)

Cautious LLMs without Neotoma in their training data were refusing the evaluation flow before reading any docs. The combination of an unknown product, a multi-sentence prompt embedding a `neotoma doctor` command, and "non-destructive test" funnel language pattern-matched to an unknown-vendor onboarding funnel.

- **Evaluation prompt reduced to one sentence (#361).** The prompt a user pastes into their agent is now `Read https://neotoma.io/evaluate and tell me whether Neotoma fits my workflow.` (Spanish: `Lee https://neotoma.io/evaluate y dime si Neotoma encaja en mi flujo de trabajo.`). All prior guidance — context sweep, status command, store-first, shadow-test offer — moved onto the `/evaluate` page, which is the single place for onboarding detail. Updated everywhere the prompt appears: the homepage evaluate card, the `/evaluate` page, integration "getting started" blocks, and `README.md`.
- **`neotoma doctor` renamed to `neotoma status` (#361).** `doctor` had no established meaning in developer CLI tooling and read as proprietary jargon. The command is now `neotoma status`, with `doctor` retained as a deprecated alias (`program.command("status").alias("doctor")`) so every existing `neotoma doctor --json` invocation keeps working. Invoking via the `doctor` alias additionally writes a one-line deprecation notice to **stderr** (stdout — including the `--json` payload — is byte-identical, so machine parsing is unaffected). The `--json` report shape is unchanged (the report's `doctor.*` keys keep their names for payload stability). The command description was rewritten from "Consolidated diagnostics for agent-led onboarding (…)" to "Check Neotoma configuration and connectivity (CLI path, data directory, local API, MCP entries, permissions)". Agent-facing and user-facing references (`install.md`, `cli_reference.md`, `cli_agent_instructions.md`, site install/evaluate copy, the `ensure-neotoma` skill) now use `neotoma status`. The developer ops health-check `npm run doctor` (`scripts/doctor.ts`) is a separate tool and is unchanged.
- **`/evaluate` page restructured to lead with facts (#362).** The page now opens with a declarative statement of what Neotoma is ("Neotoma is a deterministic state layer for AI agents…") so a model reading it cold can answer "what is this?" from the first paragraph. The "read → run → consider" funnel ordering is replaced with information-first ordering: what Neotoma is, what it stores, who it's for, the three taxes, what to store first, how it compares — and only then the assessment prompt and call to action. The page `<meta>` title/description are likewise declarative.

## Behavior changes

- **#1483:** the MCP GET SSE stream emits `: hb\n\n` comment frames (invisible to clients per the SSE spec) and carries `X-Accel-Buffering: no`.
- **#276:** callers reading `related_entities[].entity_id` now receive a value instead of `undefined`; callers still reading `related_entities[].id` continue to receive the same value (now formally deprecated).
- **#1541:** `merge_array` fields are now priority-gated. A `correct()` (or any strictly higher-priority write) on an array-typed `merge_array` field now fully **replaces** lower-priority array contributions instead of unioning with them, so corrections can cleanly reset an array. Two consequences for callers: (1) a top-priority `field: null` correction now clears such a field to `[]`; (2) `provenance[<field>]` / `source_observation_id` for a `merge_array` field now lists only the top-`source_priority` contributing observation IDs, not every observation that ever contributed an element. Same-priority observations still union as before.
- **#1595:** `merge_array` fields (and `canonicalizeArray`) now recover a JSON-array-shaped *string* (e.g. `'["a","b"]'`) into a real array instead of treating the whole blob as one literal element (reducer) or dropping it to `[]` (canonicalizer). This tolerates an upstream client/transport bug that stringifies array arguments; the Neotoma server write path itself already preserves arrays. A genuine non-JSON string is left as a single literal element; real arrays are unaffected. **Migration note:** for an instance carrying old observations that stored a stringified-array on a `merge_array` field, a snapshot recompute may now collapse `'["a","b"]'` and `["a","b"]` to the same canonical form (previously distinct); this is deterministic and idempotent and affects only convergence-comparison hashing, not stored identity/idempotency keys.
- **#1540:** `correct` no longer rejects an undeclared `field`. It accepts the correction (append path), returns `unknown_field: true` with a `hint` and `details`, and excludes the value from the snapshot until the field is declared. Callers that depended on `correct` throwing for an undeclared field must now read `unknown_field` instead. See "Breaking changes" below.
- **#1559 / #1552 / #1549:** a successful store may now carry `store_warnings[]`, `required_fields_missing[]`, and `unknown_fields` / `hint`. These are additive, non-fatal signals; existing callers that ignore them are unaffected, but agents should repair flagged fields in-turn.
- **correct transport parity:** the HTTP `/correct` handler no longer swallows a schema-registry IO failure as "declared field"; it propagates the error like the MCP path. A `correct` against a type with no active schema returns `ERR_NO_SCHEMA_FOR_ENTITY_TYPE` on HTTP, where it was previously accepted as an undeclared-field correction.
- **#361:** `neotoma status` is now the primary name for the configuration/connectivity check; `neotoma doctor` continues to work as a deprecated alias and produces identical stdout (a one-line deprecation notice is written to stderr only). `neotoma --help` now lists the command as `status|doctor`.
- **#277:** `delete_relationship` with a `(relationship_type, source_entity_id, target_entity_id)` triple that does not match a live (non-deleted) edge previously returned a silent success after writing a no-op deletion observation; it now reports a not-found failure with a discovery hint pointing to `list_relationships` for type discovery. The guard is enforced on **both** delete surfaces: the HTTP `/delete_relationship` handler returns `404 RESOURCE_NOT_FOUND` with a `details.hint`, and the MCP `delete_relationship` tool raises an `McpError` (`RESOURCE_NOT_FOUND` in its structured `data`) carrying the same hint text. A correct triple that matches a live edge still deletes and returns success as before on both surfaces. Separately, `list_relationships` now excludes soft-deleted edges by default (both the HTTP handler and the MCP tool), so after a successful `delete_relationship` the deleted edge no longer appears and a caller following the discovery-then-delete flow will not re-delete it into the new not-found failure. Pass `include_deleted: true` to include soft-deleted edges for audit/history.
- **#946 / #942:** `submit_issue` / `POST /issues/submit` accept an optional `target_repo`. Callers that omit it see identical behavior to prior releases (mirror to `NEOTOMA_ISSUES_REPO`). Callers that set it get the GitHub mirror in the named repo and an `issue` / `conversation` identity namespaced to it. A malformed `target_repo` is now rejected with an `owner/repo` validation error, where previously such a value would have flowed through and (on the GitHub leg) silently failed the mirror. **Operator-side caveat:** the operator instance's `submitIssueToRemote` cross-instance forwarder does not propagate `target_repo`; the override governs the submitter-side GitHub mirror and the submitter's local `issue` identity. This is intentional for the single-operator multi-repo case but means a submitter and a forwarded operator copy can namespace identity to different repos — documented for the forthcoming cross-maintainer routing work (#1492).

## Fixes

- #1483 — Neotoma MCP SSE connection drops under load. Added an application-level heartbeat (`: hb\n\n` comment frames), `X-Accel-Buffering: no`, and a bounded TCP keepalive to the MCP StreamableHTTP transport's GET SSE stream so it survives proxy and client idle timeouts without manual reconnection. This is the application-level layer above the v0.13.0 `keepAliveTimeout` fix (#148), which addressed socket reuse but not SSE stream idle. The `NEOTOMA_MCP_SSE_KEEPALIVE_MS` disable knob parses defensively so a literal `0` (or negative) value disables the heartbeat rather than silently falling back to the default.
- #276 — `retrieve_graph_neighborhood` `related_entities` exposed only the raw `id`; now exposes the canonical `entity_id`, with `id` retained as a deprecated alias.
- #1541 — `correct()` on an array-typed (`merge_array`) field unioned the correction into the prior array instead of replacing it; the reducer now priority-gates the `merge_array` union so corrections replace lower-priority arrays.
- #1595 — a JSON-array-shaped string delivered to a `merge_array` field (an upstream client/transport stringification of an array argument) surfaced as a single literal-string element. The reducer and `canonicalizeArray` now defensively parse such a string back into a real array via a shared `recover_json_array_string` helper. The Neotoma server write path itself was verified clean (it preserves arrays); this fix is tolerance of the external cause.
- #247 — no first-class way to introspect a single entity type's declared field shape before storing; agents fell back to `list_entity_types` keyword search or retrieving a sample entity. Added the `describe_entity_type` MCP tool returning the full active schema.
- #1559 — required fields were silently unenforced: a store that omitted a `required: true` field returned success with no signal. The store now emits `MISSING_REQUIRED_FIELD` warnings and a `required_fields_missing[]` array so agents repair in-turn.
- #1552 — undeclared fields on a store were dropped to `raw_fragments` with only `unknown_fields_count`; added a per-field `UNKNOWN_FIELD` `store_warnings[]` entry so the signal is switchable by code.
- #1549 — the `unknown_fields` repair hint blindly prescribed `update_schema_incremental`, which dead-ends on a schema with no `canonical_name_fields` (`ERR_SCHEMA_MISSING_IDENTITY_CONFIG`). The hint is now conditional on the schema's identity configuration and points at the path that works.
- #1540 — `correct` rejected an undeclared field instead of appending it; it now mirrors `store`'s append path and returns `unknown_field: true`. The MCP and HTTP `correct` responses are unified via a shared `buildCorrectionResponse` / `resolveCorrectionSchema`, closing two divergences: HTTP previously swallowed a schema-registry IO failure as "declared field" (violating product principle 10.2), and the two transports returned different envelopes.
- #361 — cold-start LLMs refused the evaluation flow before reading docs (unknown product + multi-sentence prompt + `doctor` jargon + "non-destructive test" funnel framing). Reduced the evaluation prompt to one sentence + link, renamed `neotoma doctor` → `neotoma status` (deprecated `doctor` alias retained), and rewrote the command description to plain language.
- #362 — the `/evaluate` page's "read → run → consider" structure read as a conversion funnel to a cautious LLM. Restructured to lead with a declarative statement of what Neotoma is and to order content information-first, with the assessment prompt and CTA following the facts.
- #277 — `delete_relationship` required the exact `relationship_type` between two entities but offered no discovery path, and a wrong guess silently "succeeded" by writing a no-op deletion observation for an edge that never existed. Both delete surfaces now verify a live edge matches the supplied triple before recording the deletion: the HTTP handler returns `404 RESOURCE_NOT_FOUND` with a structured `details.hint`, and the MCP `delete_relationship` tool raises an `McpError` whose `data` carries `code: "RESOURCE_NOT_FOUND"` and the same hint, both steering the caller to `list_relationships` for type discovery. (The guard is mirrored into each handler rather than pushed into `softDeleteRelationship`, because the GDPR bulk-deletion path intentionally writes a deletion tombstone for every observed edge.) To make that discovery flow self-consistent, `list_relationships` (HTTP and MCP) now filters soft-deleted edges by default with an `include_deleted` opt-in, so a deleted edge is not re-offered for deletion.
- #1570 — the default `list_relationships` soft-delete filter (#277) loaded the full matching edge set into memory before filtering and slicing in process (reverting the DB-side `O(limit)` pagination shipped for #367 on the default, common path; the MCP tool was worse, running two unbounded `select('*')` queries). Liveness is now **materialized** as a `relationship_snapshots.is_live` column, derived from the highest-priority deletion observation per `relationship_key` — the same rule `filterDeletedRelationships` applies — and re-stamped on every snapshot recompute and on `softDeleteRelationship`. The default read filters `is_live = 1` as a DB predicate and paginates at the DB (`.range()` + exact `count`), restoring `O(limit)` memory on the hot path. `include_deleted: true` keeps the unfiltered audit path. A one-time, idempotent backfill reconciles existing snapshots from the observation log on schema init. Internal change only — no `openapi.yaml` surface; `is_live` is not exposed in any response.
- #1571 — the MCP `list_relationships` tool sorted by `last_observation_at DESC` with no tiebreaker, so rows sharing a `last_observation_at` (batch upserts) could move between pages, dropping or duplicating rows across `offset`/`limit` pages. The MCP sort now tiebreaks on `relationship_key ASC`, matching the HTTP handler and the determinism invariant (#368).

## Breaking changes

Two `correct` behavior changes in the #1540 cluster are flagged here:

1. **`correct` against an `entity_type` with no active schema now errors on HTTP.** Previously the HTTP `/correct` handler treated "no schema found" as an undeclared-field correction and accepted the write. It now returns `ERR_NO_SCHEMA_FOR_ENTITY_TYPE` (HTTP 400), matching the MCP `correct` tool's pre-existing `InvalidParams` throw. This unifies the two transports' unknown-field determination and removes the path where an HTTP `correct` silently succeeded against a non-existent type. Callers must register the schema (`register_schema`) before correcting, or read the error. This is the only narrowing of a previously-accepted `correct` request shape in this release.

2. **`correct` against an undeclared field no longer throws — it now succeeds with `unknown_field: true`.** This is a widening (previously-rejected input is now accepted), not a narrowing, but it changes observable behavior: callers that relied on `correct` throwing "Unknown field for entity type" for an undeclared field will no longer see that error. They must switch on the `unknown_field` flag in the (now 2xx) response instead. The `correct` response envelope is unified across MCP and HTTP via `buildCorrectionResponse`; HTTP additionally returns `success` and `snapshot`.

3. **`POST /correct [200]` response `additionalProperties` tightened to `false`** (`openapi:bc-diff` change kind `tightened-additional-properties`). The `CorrectResponse` schema now declares every field the handler returns (`observation_id`, `entity_id`, `field`, `value`, `message`, `success`, `snapshot`, `unknown_field`, `hint`, `details`) and closes the shape. Migration note: the handler never returned fields outside this set, so no real response is narrowed; this only affects clients whose schema-validation tooling asserted against the previously-open response. No request shape is tightened.

The `neotoma doctor` → `neotoma status` rename (#361) is **not** classified as breaking: `doctor` is retained as a deprecated alias that resolves to the identical handler and `--json` report shape, so every existing `neotoma doctor`/`neotoma doctor --json` invocation continues to work unchanged (the only added behavior is a one-line stderr deprecation notice; stdout, including the `--json` payload, is byte-identical). The alias will be removed in a future minor release after callers migrate; that removal will be flagged as breaking at the time. No request shape, response shape, or report key was narrowed or removed.

No other breaking changes. Re-adding the `related_entities[].id` alias (#276) is additive; the field is now marked deprecated and will be removed in a future minor release after callers migrate to `entity_id`. The new store-response signals (`store_warnings[]`, `required_fields_missing[]`, `unknown_fields`, `describe_entity_type`) are additive and optional; callers that ignore them are unaffected. The `submit_issue` `target_repo` parameter (#946 / #942) is additive and optional: it adds a new optional request field with default-unset behavior identical to prior releases, and its `owner/repo` validation applies only to the new field — no previously-accepted request shape is narrowed.

The #1541 `merge_array` changes listed under "Behavior changes" are **not** classified as breaking: they correct `correct()` to its documented semantics (corrections "always win"). (a) The provenance-shape change only affects code that walks `provenance[<field>]` / `source_observation_id` for a `merge_array` field on a correction code path — non-correction paths are unaffected, and the documented contract was always "the contributing source(s)", not "every observation that ever touched the field"; callers should walk only the IDs listed and not assume every contributor appears. (b) The null-correction-clears-to-`[]` behavior is a new capability (previously there was no single-call way to clear such a field), not a narrowing of any previously-accepted input. No API contract is narrowed and no field is removed.

The #277 `delete_relationship` change listed under "Behavior changes" is **not** classified as breaking. The prior `200` response was a silent no-op: it wrote a deletion observation for an edge that never existed and deleted nothing, so no caller depended on a useful contract there. The new `404 RESOURCE_NOT_FOUND` makes the existing failure explicit; a correct triple that matches a live edge behaves exactly as before. The HTTP status code observably changed (`200` → `404`) only for the wrong-/unknown-type case, so it is called out here explicitly. The `list_relationships` soft-delete filtering is additive (it removes rows that were never useful deletion targets) and gated by the new opt-in `include_deleted` flag, which defaults to the prior-intent (live-only) view; no request shape is narrowed and no response field is removed.
